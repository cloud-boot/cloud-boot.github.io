A Go-native multi-host cloud platform — workloads flow across the mesh.

weft is the control plane plus per-host agents that run virtual machines on a mesh of compute hosts. The control plane owns the data — projects, users, networks, volumes, security groups, ports, hosts, VM inventory — and the agents own the drivers that talk to the hypervisor, the kernel, and the storage backend. The shuttle moves between the nodes; the warp stays put.

See the architecture GitHub

Hosts are nodes ; workloads (spheres) cross the mesh via the control-plane dispatch table. Hub nodes carry the heaviest traffic.

Why weft

VMs are stateful, hosts fail, drivers crash, and operators don't want any of that to show through. weft is the smallest set of components that lets a workload land on the right host, talk to the right driver, and survive an agent reconnect — without a kernel module on the host or a third-party agent fleet.

Control plane stays small

Eight UUID-keyed registries (projects, users, networks, volumes, security groups, ports, hosts, VMs) backed by etcd in production, by HCL files in dev. No business logic in storage. Cross-registry validation centralised in the Adapter.

Drivers are interfaces first

Every side-effecting bit — VM lifecycle, network plumbing, storage allocation, OCI image cache — sits behind a Go interface in weft-drivers. builtin for in-process today, gRPC plugins tomorrow, third-party implementations whenever.

Mesh dispatch, not RPC fan-out

Each agent attaches its driver tuple to the control plane's dispatch table under its host UUID. The scheduler picks a host ; the dispatcher hands the call to that host's drivers ; the result returns over the same stream. Same code path for one host or fifty.

Where the metaphor comes from

Weft is the crosswise thread in weaving — the shuttle that moves through the stationary warp and builds the fabric one pass at a time. That's the platform's mental model in one word :

Warp — what stays put

  • The control plane and its registries.
  • The host inventory and the dispatch table.
  • etcd, NATS, dex, zot — the infra services that live as micro-VMs on weft.
  • The driver interfaces : a v1 contract that survives multiple driver implementations.

Weft — what crosses

  • VM lifecycle calls dispatched from the control plane down to the right host's drivers.
  • Heartbeats up, driver-call results down, event-bus messages everywhere.
  • The scheduler picking a host per request — first-fit for one VM, anti-affinity for a 3-replica group.
  • Workloads migrating between hosts when one drains.

The three roles

Three binaries, one cluster. Each role has one job ; together they form the loop that creates, starts, stops, and migrates VMs across the mesh.

weft

Control plane

The central daemon. Owns the registries via the Storage interface ; serves the public gRPC API ; runs the scheduler ; routes lifecycle calls through the dispatch table. Stateless beyond etcd.

  • Registries : projects, users, networks, volumes, SGs, ports, hosts, VMs
  • Scheduler with PlacementRule (anti-affinity AZ / Host)
  • Event bus : LocalEventBus for dev, NATS JetStream for prod
  • OIDC validation via dex ; per-project ACLs
vzd-agent

Per-host agent

One process per compute node. Builds the local driver bundle, registers with the control plane under its persisted host UUID, then serves driver calls over a long-lived bidirectional stream.

  • Persisted identity at <state-dir>/host-uuid
  • RegisterAgent → AttachDrivers → heartbeat loop
  • Owns the local driver Bundle from weft-driver-vz/builtin
  • Reconciler aligns local state with the control plane's intent
weft-driver-*

Drivers

Per-resource, per-platform implementations of the four interfaces (Hypervisor, Network, Volume, Image). Each driver lives in its own module ; eventually each in its own repo so release cadence is independent.

  • weft-driver-vz — Apple Virtualization.framework (macOS)
  • weft-driver-qemu-kvm — Linux x86_64 / arm64 (planned)
  • weft-driver-wireguard — mesh-type networks (planned)
  • weft-driver-ceph, -iscsi, -file — volumes

Repos & modules

Today everything lives in one monorepo ; the eventual layout under github.com/cloud-boot is one repo per driver so vendors and operators can pin and patch independently.

cloud-boot — the sibling project

cloud-boot is the boot toolchain that complements weft. It's a tiny UEFI bootloader plus a UKI assembler that lets weft boot unmodified Linux, FreeBSD, NetBSD, OpenBSD cloud images — and Windows via DevicePath handoff — without rebuilding them. One multi-arch hybrid ISO covers x86_64, arm64, riscv64 and loongarch64.
